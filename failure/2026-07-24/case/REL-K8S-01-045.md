## 失败分诊 · REL-K8S-01-045 · 自托管 K8s Runner 弹性伸缩——min=1/max=1 时并发 3 个 jobs 应排队执行

**判定结果**: FAIL
**失败断言**: assertions[0] (positive, run_status) — 期望 job 执行并输出 Pod 数和并发信息，实际 job status=FAILED，0 字节有效日志

**根因初判**: 环境问题

**证据**:

- **Job 日志全量**（仅 2 行）:
  ```
  === JOB: K8s runner scaling test (status=FAILED) ===
  [2026/07/23 22:30:25.434 GMT+08:00] [INFO] Job(1529979023189090304_1529979023155535879) duration check: true
  ```
  日志显示：job 状态 **FAILED**，但**仅有 2 行 header 行**——无任何 shell 步骤输出、无 `No shell specified`、无 `Script file created`、无 `Executing` 行。这是最极端的"无有效日志"情况。self-hosted K8s runner 从未启动执行——platform 调度失败，可能原因是 K8s runner 标签 (`labels: self-hosted-k8s`) 找不到可用 runner 实例，或 K8s runner 未注册/离线。

- **预期行为**（Phase 01 文本用例 `REL-K8S-01-045`，优先级 P1，维度 稳定性）:
  - 操作步骤 1: "并发触发 3 个 jobs 到 K8s Runner 组"
  - 预期结果: "Runner Pod 数始终=1；3 个 jobs 顺序执行；不应创建 2 个及以上 Pod"
  - 验证点: "[正向] Pod 数=1；[正向] 峰值并发=1；[负向] 不应创建 2 个及以上 Pod"

- **实际行为**:
  - job 创建后从未开始执行——runner 未被调度到
  - 日志中无任何 shell 执行证据、无 bash 错误、无 runner 连接日志
  - 0 字节有效日志说明 runner 环境完全不可用

- **测试 YAML 与规格精确对照**:
  - 测试 YAML 中 `K8s runner scaling test` job 配置 `runs-on: self-hosted-k8s` 标签
  - 这对应 GitCode 规格 `phase01/inputs/gitcode-spec/core-concepts/runner-and-environment.md` 中 self-hosted runner 的注册和调度行为。规格描述 self-hosted runner 通过标签匹配 job——当 runner 离线或无匹配时，job 将进入 pending/queued 状态，最终超时失败（status=FAILED）。**平台正确标记了 job 为 FAILED，但 self-hosted K8s runner 未就绪**。

**置信度**: 中（0 字节有效日志——K8s runner 未启动或未匹配到标签 `self-hosted-k8s`，属测试环境基础设施问题，非平台功能缺陷）

**建议**:
- 检查 K8s runner 组是否已注册并与 `self-hosted-k8s` 标签关联
- 验证 K8s runner Pod 是否在线（`kubectl get pods` 检查 runner namespace）
