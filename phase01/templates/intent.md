# Test Intent 模板

> 阶段 A 各维度 agent 产出。intent 只描述意图层：测什么 / 为什么有风险 / 预期系统行为。**不写执行细节、不写 GitCode 具体语法。**
> 每条 intent 用下面的字段块。多条并列。

```
意图 ID:    INTENT-<维度>-<序号>      # 维度: COMP|COMPAT|REL|SEC|USE
维度标签:   [<completeness|compatibility|reliability|security|usability>]  # 可多值
标题:       <一句话说清要验证什么>

风险点:     <为什么这里有风险 / 会出什么问题>
预期系统行为: <系统正确时应表现出的行为>
Oracle 来源: <GitCode规格 | GitHub行为 | 差异声明>   # 见 rules.md §4
             # 兼容性 intent 必须写清「对齐谁」：一致性 / 差异确认

验证要点:
  - [正向] <应发生什么>
  - [负向] <什么不应发生>            # 安全 intent 必含
  - [非功能] <时序/并发/可理解性>    # 按需

# —— 以下为各维度补充字段（按维度填相应项）——
对齐方向:   <一致性 | 差异确认>       # 仅兼容性
负向断言目标: <什么绝不应发生 + 判定证据>  # 仅安全
故障/压力参数: <并发度/规模/注入时机与类型/恢复预期>  # 仅稳定性
可理解性判据: <错误信息应包含什么；是否 llm_assisted>  # 仅易用性

优先级线索: <关联哪个 RISK-xxx>       # 门禁据此定级，agent 给线索不拍板
破坏级别:   <fixture | full_instance | none>  # 破坏性 intent 必填
来源输入:   <依据哪份 input；版本/日期；若缺输入在此标注>
```

## 填写要点
- 一条 intent = 一个风险点/一个待验证行为，别把多个塞一条。
- 兼容性、安全、稳定性、易用性各自的专属字段务必填（见上）。
- 无法对齐任何风险项的 intent，门禁可能打回——先在「优先级线索」诚实标注。
